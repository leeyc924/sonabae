# PRD: 소나배 2차 개선

- 문서 버전: v2.0
- 작성일: 2026-05-11
- 제품명: 소나배
- 제품 유형: 모바일 앱
- 기술 스택: Expo React Native, TypeScript, AsyncStorage 기반 로컬 저장소
- 기준 문서: `docs/SONABAE_PRD.md`, `DESIGN.md`
- 현재 범위: MVP 기능 확장 및 사용자 피드백 반영

---

## 1. 제품 개요

### 1.1 목적

소나배 2차 버전은 MVP에서 확인된 핵심 사용성 문제를 개선하고, 기록 데이터의 정확도를 보완하는 것을 목표로 한다.

이번 버전의 중심은 신규 기능의 대규모 추가가 아니라 사용자가 운동 후 기록을 헷갈리지 않고 저장할 수 있게 만드는 것이다. 특히 기록, 관리, 상세보기, 승률 계산의 신뢰도를 우선 개선한다.

### 1.2 2차 개선 한 줄 정의

**기록과 관리 화면을 명확히 분리하고, 저장 피드백과 상세보기, 정확한 승패/승률 계산을 제공하는 소나배 사용성 개선 버전.**

---

## 2. 배경 및 문제 정의

### 2.1 MVP 이후 확인된 문제

MVP는 운동일지, 사람, 모임, 대회, 급수, 승률 통계를 기록할 수 있는 기본 구조를 제공한다. 그러나 실제 사용 관점에서는 다음 문제가 확인되었다.

1. 기록 화면의 등록/수정 UI와 목록 UI가 섞여 있어 사용자가 현재 무엇을 하는 중인지 헷갈린다.
2. 관리 화면에서도 등록/수정 UI와 목록 UI가 분리되어 있지 않아 관리 흐름이 불명확하다.
3. 날짜 선택이 직관적이지 않으며, 달력 기반 선택이 필요하다.
4. 장소를 매번 텍스트로 입력하는 방식은 재사용성과 관리성이 떨어진다.
5. 저장 성공/실패에 대한 피드백이 부족해 사용자가 저장 여부를 확신하기 어렵다.
6. 기록 상세보기 화면이 없어 저장된 기록의 세부 내용을 다시 확인하기 어렵다.
7. 승패 및 승률 계산에 오류가 있으며, 나와 파트너가 이겼는데 상대 팀도 이긴 것으로 표시되는 문제가 있다.
8. 급수 입력이 자유 입력 중심이면 값이 흔들리고 통계/필터 기준으로 사용하기 어렵다.

### 2.2 핵심 문제

**사용자가 기록과 관리를 수행하는 화면 구조가 헷갈리고, 저장된 경기 결과 및 승률 계산을 신뢰하기 어렵다.**

## 3. 목표와 비목표

### 3.1 제품 목표

1. 기록 목록, 등록, 수정, 상세보기를 명확히 분리한다.
2. 관리 목록, 등록, 수정 화면을 명확히 분리한다.
3. 날짜, 장소, 급수 등 반복 입력 항목을 선택형 UI로 개선한다.
4. 저장 성공/실패 상태를 즉시 인지할 수 있게 한다.
5. 승패 및 승률 계산 기준을 명확히 정의하고 오류를 수정한다.

### 3.2 성공 기준

| 지표 | 목표 |
|---|---|
| 기록 저장 완료 인지 | 저장 후 성공/실패 토스트가 항상 표시됨 |
| 기록 상세 접근성 | 목록에서 특정 기록 상세로 진입 가능 |
| 등록/수정 혼동 감소 | 목록 화면과 폼 화면이 별도 화면으로 분리됨 |
| 승패 계산 정확도 | 내 팀 승리 시 상대 팀이 승리로 표시되지 않음 |
| 급수 데이터 일관성 | 급수 입력은 기본적으로 selectbox 선택값으로 저장됨 |

### 3.3 비목표

이번 2차 범위에서는 다음을 다루지 않는다.

- 소셜 로그인
- 클라우드 동기화
- 결제
- 광고
- 커뮤니티
- 채팅
- AI 분석 리포트
- 모임원 간 기록 공유
- 대회 접수 기능

## 6. 기능 상세 요구사항

## V2-FR-001. 기록 목록/등록/수정 화면 분리

### 설명

기록 화면은 목록, 등록, 수정 역할을 한 화면에서 동시에 처리하지 않는다. 사용자는 먼저 기록 목록을 보고, 새 기록 작성 또는 기존 기록 수정 시 별도 폼 화면으로 이동한다.

### 화면 구조

| 화면 | 역할 |
|---|---|
| 기록 목록 | 저장된 운동일지 목록 확인, 상세보기 진입, 새 기록 버튼 |
| 기록 등록 | 새 운동일지 작성 |
| 기록 수정 | 기존 운동일지 수정 |
| 기록 상세 | 저장된 기록의 세부 내용 확인 |

### UX 요구사항

- 기록 목록 상단에는 “새 기록” 진입 버튼을 제공한다.
- 기록 등록 화면은 저장 전 임시 입력 상태를 유지한다.
- 기록 수정 화면은 기존 값을 모두 채운 상태로 시작한다.
- 저장 완료 후 사용자는 기록 상세 또는 기록 목록으로 이동한다.
- 목록 화면에서는 입력 폼을 직접 노출하지 않는다.

### 완료 조건

- 사용자는 목록을 보다가 명확한 버튼을 통해 등록 화면으로 진입할 수 있어야 한다.
- 사용자는 상세 화면에서 수정 화면으로 진입할 수 있어야 한다.
- 목록, 등록, 수정 화면의 제목과 CTA가 서로 구분되어야 한다.

---

## V2-FR-002. 기록 상세보기

### 설명

사용자는 저장된 운동일지의 날짜, 경기 구분, 장소, 연결된 모임/대회, 경기 목록, 점수, 승패, 메모를 상세 화면에서 확인할 수 있어야 한다.

### 노출 정보

- 날짜
- 경기 구분: 모임 또는 대회
- 모임명 또는 대회명
- 장소
- 메모
- 경기 목록
- 각 경기의 경기 유형: 단식 또는 복식
- 파트너
- 상대
- 점수
- 내 결과: 승리, 패배, 결과 미정
- 경기별 메모

### 상세 화면 액션

- 수정
- 삭제
- 목록으로 돌아가기

### 완료 조건

- 기록 목록의 각 항목을 누르면 상세 화면으로 이동해야 한다.
- 상세 화면에서 경기별 점수와 승패를 확인할 수 있어야 한다.
- 상세 화면에서 표시되는 승패는 통계에 사용되는 결과와 일치해야 한다.

## V2-FR-003. 경기 구분 정리

### 설명

2차 버전에서 기록의 경기 구분은 `모임`과 `대회` 2개만 제공한다.

### 정책

- 기존 MVP의 `연습` 구분은 2차 주요 입력 UI에서는 제거한다.
- 사용자가 `모임`을 선택하면 모임 선택 필드를 노출한다.
- 사용자가 `대회`를 선택하면 대회 선택 필드를 노출한다.
- 선택한 구분에 맞지 않는 필드는 저장하지 않는다.

### 데이터 정책

```ts
type MatchContext = 'MEETING' | 'TOURNAMENT';
```

기존 데이터에 `PRACTICE`가 존재할 수 있다면 마이그레이션 또는 하위 호환 표시 정책을 별도로 둔다.

### 완료 조건

- 기록 등록/수정 화면에서 경기 구분은 `모임`, `대회` 2개만 선택 가능해야 한다.
- 경기 구분 변경 시 연결 대상 선택 UI가 즉시 변경되어야 한다.

---

## V2-FR-004. 달력 날짜 선택

### 설명

날짜 입력은 텍스트 직접 입력이 아니라 달력 UI를 통해 선택한다.

### 요구사항

- 기본값은 오늘 날짜로 설정한다.
- 날짜 필드를 누르면 달력 선택 UI가 열린다.
- 선택 후 날짜는 `YYYY-MM-DD` 형식으로 저장한다.
- 기록 등록, 기록 수정, 대회 등록/수정 등 날짜 입력이 필요한 화면에 동일한 패턴을 적용한다.

### 완료 조건

- 사용자는 직접 날짜 문자열을 입력하지 않고 날짜를 선택할 수 있어야 한다.
- 잘못된 날짜 형식으로 저장되는 일이 없어야 한다.

---

## V2-FR-005. 장소 관리

### 설명

장소는 기록 작성 시 매번 텍스트로 입력하는 값이 아니라 등록 및 관리 가능한 데이터로 다룬다.

### 장소 데이터

```ts
type Place = {
  id: string;
  userId: string;
  name: string;
  address?: string;
  memo?: string;
  createdAt: string;
  updatedAt: string;
};
```

### 화면 구조

| 화면 | 역할 |
|---|---|
| 장소 목록 | 등록된 장소 확인 |
| 장소 등록 | 새 장소 등록 |
| 장소 수정 | 기존 장소 수정 |

### 기록 연동

- 기록 등록/수정 화면에서 장소를 선택할 수 있어야 한다.
- 장소가 없을 경우 기록 화면에서 장소 등록 화면으로 진입할 수 있어야 한다.
- 기존 `location` 문자열은 하위 호환을 위해 유지할 수 있으나, 신규 저장은 `placeId`를 우선 사용한다.

### 완료 조건

- 사용자는 장소를 등록, 수정, 삭제할 수 있어야 한다.
- 기록 작성 시 등록된 장소를 선택할 수 있어야 한다.
- 장소 삭제 시 기존 기록의 표시 정책이 정의되어야 한다. 예: `삭제된 장소`.

## V2-FR-006. 저장 성공/실패 토스트

### 설명

등록, 수정, 삭제 등 저장성 액션 이후에는 성공 또는 실패 상태를 토스트로 표시한다.

### 적용 범위

- 기록 등록
- 기록 수정
- 기록 삭제
- 사람 등록/수정/삭제
- 모임 등록/수정/삭제
- 대회 등록/수정/삭제
- 장소 등록/수정/삭제
- 프로필/급수 수정

### 메시지 예시

| 상황 | 메시지 |
|---|---|
| 기록 저장 성공 | 기록을 저장했어요. |
| 기록 저장 실패 | 기록 저장에 실패했어요. 다시 시도해 주세요. |
| 수정 성공 | 변경사항을 저장했어요. |
| 삭제 성공 | 삭제했어요. |
| 삭제 실패 | 삭제에 실패했어요. 다시 시도해 주세요. |

### UX 정책

- 토스트는 화면 하단 또는 상단 중 앱의 기존 UI 패턴에 맞춰 일관되게 노출한다.
- 성공 토스트와 실패 토스트는 색상뿐 아니라 텍스트로도 구분한다.
- 실패 시 사용자가 재시도할 수 있도록 현재 입력값을 유지한다.

### 완료 조건

- 저장성 액션 이후 성공/실패 토스트가 항상 표시되어야 한다.
- 실패 시 입력 중인 내용이 사라지지 않아야 한다.

## V2-FR-007. 관리 화면 구조 개선

### 설명

관리 화면의 사람, 모임, 대회, 장소 관리는 목록과 등록/수정 화면을 분리한다.

### 관리 대상

- 사람
- 모임
- 대회
- 장소

### 화면 구조

| 화면 | 역할 |
|---|---|
| 관리 홈 | 사람, 모임, 대회, 장소 관리 섹션 진입 |
| 각 관리 목록 | 대상 데이터 목록 확인 |
| 각 등록 화면 | 새 데이터 등록 |
| 각 수정 화면 | 기존 데이터 수정 |

### UX 요구사항

- 목록 화면에 입력 폼을 직접 노출하지 않는다.
- 목록 항목 선택 시 상세 또는 수정 화면으로 이동한다.
- 신규 등록은 명확한 `추가` 버튼을 통해 진입한다.
- 저장 후 성공/실패 토스트를 표시한다.

### 완료 조건

- 사용자는 목록 화면과 등록/수정 화면을 혼동하지 않아야 한다.
- 각 관리 대상은 동일한 화면 패턴을 사용해야 한다.

## V2-FR-008. 급수 selectbox

### 설명

급수는 자유 입력이 아니라 selectbox로 선택한다.

### 기본 급수 옵션

- 초심
- D조
- C조
- B조
- A조
- S조
- 기타

### 적용 범위

- 초기 프로필 설정
- 내 정보 수정
- 사람 등록/수정
- 대회 등록/수정
- 급수 변경 이력

### 정책

- 기본 입력은 selectbox로 제공한다.
- `기타` 선택 시 사용자 정의 텍스트 입력을 허용할 수 있다.
- 저장되는 값은 통계와 필터에 사용할 수 있도록 일관된 문자열이어야 한다.

### 완료 조건

- 급수 입력 필드는 selectbox로 제공되어야 한다.
- 동일한 급수가 화면마다 다른 값으로 저장되지 않아야 한다.

## V2-FR-009. 승패 및 승률 계산 수정

### 설명

나와 파트너가 이긴 경기에서 상대 팀도 이긴 것으로 표시되는 오류를 수정한다. 승패는 항상 “사용자 기준 결과”와 “팀 기준 결과”를 구분해서 계산한다.

### 핵심 원칙

1. `Match.result`는 사용자 기준 결과다.
2. 나와 같은 팀의 파트너는 사용자와 같은 결과를 가진다.
3. 상대 팀 선수는 사용자와 반대 결과를 가진다.
4. 상대 팀이 이긴 것으로 표시되는 경우는 사용자가 패배한 경기뿐이다.
5. 승률 통계는 사용자 기준 결과를 사용한다.

### 점수 기반 승패 계산

```text
if myTeam = 'A' and scoreA > scoreB => WIN
if myTeam = 'A' and scoreA < scoreB => LOSS
if myTeam = 'B' and scoreB > scoreA => WIN
if myTeam = 'B' and scoreB < scoreA => LOSS
if scoreA = scoreB or score missing => UNKNOWN
```

### 사람별 결과 표시

| 사람 관계 | 사용자 결과가 WIN일 때 | 사용자 결과가 LOSS일 때 |
|---|---|---|
| 파트너 | 승리 | 패배 |
| 상대 | 패배 | 승리 |

### 사람별 승률 계산

파트너 승률은 해당 사람이 나와 같은 팀으로 뛴 경기에서 사용자 기준 승리 비율을 계산한다.

```text
파트너 승률 = 해당 사람과 같은 팀으로 뛴 경기 중 사용자 승리 수 / 해당 사람과 같은 팀으로 뛴 결과 확정 경기 수 * 100
```

상대 전적은 해당 사람이 상대 팀으로 뛴 경기에서 사용자 기준 승리 비율을 계산한다.

```text
상대 전적 승률 = 해당 사람이 상대 팀으로 뛴 경기 중 사용자 승리 수 / 해당 사람이 상대 팀으로 뛴 결과 확정 경기 수 * 100
```

단, 상대 선수 상세에서 “상대의 결과”를 직접 표시할 경우 사용자 결과의 반대값으로 표시한다.

### 테스트 케이스

| 케이스 | 조건 | 기대 결과 |
|---|---|---|
| 내 팀 A 승리 | `myTeam=A`, `scoreA=21`, `scoreB=15` | 사용자 WIN, 파트너 WIN, 상대 LOSS |
| 내 팀 A 패배 | `myTeam=A`, `scoreA=18`, `scoreB=21` | 사용자 LOSS, 파트너 LOSS, 상대 WIN |
| 내 팀 B 승리 | `myTeam=B`, `scoreA=15`, `scoreB=21` | 사용자 WIN, 파트너 WIN, 상대 LOSS |
| 내 팀 B 패배 | `myTeam=B`, `scoreA=21`, `scoreB=18` | 사용자 LOSS, 파트너 LOSS, 상대 WIN |
| 동점 | `scoreA=20`, `scoreB=20` | UNKNOWN |
| 점수 없음 | 점수 미입력 | UNKNOWN 또는 수동 선택값 |

### 완료 조건

- 내 팀이 승리한 경기에서 상대 팀이 승리로 표시되지 않아야 한다.
- `myTeam`이 A/B 어느 쪽이어도 결과가 정확해야 한다.
- 전체 승률, 사람별 승률, 모임별 승률, 대회별 승률이 동일한 계산 기준을 사용해야 한다.
- 위 테스트 케이스를 기준으로 수동 검증 또는 단위 테스트를 작성해야 한다.

## 9. 데이터 모델 변경안

### 9.1 MatchContext 변경

```ts
type MatchContext = 'MEETING' | 'TOURNAMENT';
```

기존 데이터 호환을 위해 내부 타입에는 `PRACTICE`를 임시 허용할 수 있으나, 신규 입력 UI에서는 노출하지 않는다.

  name: string;

  updatedAt: Timestamp;

### 9.3 Session 변경

```ts
type Session = {
  id: ID;
  userId: ID;
  date: string;
  context: 'MEETING' | 'TOURNAMENT';
  meetingId?: ID;
  tournamentId?: ID;
  placeId?: ID;
  location?: string;
  memo?: string;
  createdAt: Timestamp;
  updatedAt: Timestamp;
};
```

### 9.4 Match 계산 필드 정책

```ts
type Match = {
  id: ID;
  userId: ID;
  sessionId: ID;
  meetingId?: ID;
  tournamentId?: ID;
  matchType: 'SINGLES' | 'DOUBLES';
  context: 'MEETING' | 'TOURNAMENT';
  myTeam: 'A' | 'B';
  teamA: ID[];
  teamB: ID[];
  scoreA?: number;
  scoreB?: number;
  result: 'WIN' | 'LOSS' | 'UNKNOWN';
  memo?: string;
  playedAt: string;
  createdAt: Timestamp;
  updatedAt: Timestamp;
};
```

`result`는 저장 시점 또는 통계 계산 시점에 `myTeam`, `scoreA`, `scoreB`를 기준으로 산출한다. 수동 선택을 허용할 경우에도 “사용자 기준 결과”로 저장한다.

## 13. 최종 2차 정의

**소나배 2차 버전은 MVP의 기록 가능 상태를 넘어, 사용자가 헷갈리지 않고 기록을 작성/수정/확인하며, 저장 결과와 승률 계산을 신뢰할 수 있게 만드는 UX 및 데이터 정확도 개선 버전이다.**

2차에서 반드시 지켜야 할 기준은 다음 세 가지다.

1. 목록과 등록/수정 화면은 분리한다.
2. 저장 성공/실패는 사용자에게 즉시 알려준다.
3. 승패와 승률은 사용자 팀 기준으로 정확하게 계산한다.
